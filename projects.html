AI-Driven Cardiovascular Disease
Risk Prediction System

A final-year capstone that fuses a trained ML risk classifier with an offline, guideline-grounded RAG chatbot — so a prediction never arrives without an explanation rooted in real clinical literature.

5Agentic tools
4Guideline sources
2Ports, coordinated

The problem

Risk calculators give a number. They don't explain themselves, and they're rarely grounded in the guidelines a clinician would actually cite. The goal was a system that does both — predicts CVD risk from patient data and answers follow-up questions with responses traceable to AHA/ACC, ESC, WHO, and NCEP guidance, entirely offline.

ML pipeline

Trained and compared Logistic Regression (primary) against Random Forest (backup) on the UCI Cleveland Heart Disease dataset. Early runs overfit; resolved through dataset deduplication, cleaning, and feature selection. Evaluated with ROC-AUC and cross-validation rather than raw accuracy, given the clinical cost of false negatives.

RAG + agentic layer

The chatbot runs fully offline: LangChain orchestrates retrieval over a ChromaDB vector store, HuggingFace embeddings index the guideline corpus, and Llama 3.2 generates responses locally via Ollama — no cloud API in the loop. A five-tool agentic layer sits in front of generation, routing each patient query to the correct guideline domain (hypertension, CHD, general risk, lifestyle, emergency triage) before the model responds, instead of dumping the entire knowledge base into context every time.

Deployment

Both services run as coordinated multi-service backends behind FastAPI — ML inference on port 8000, the RAG agent on port 8001. A Streamlit web frontend merges the risk score and the chatbot's grounded explanation into a single result view; a parallel Flutter mobile app (MVVM + Riverpod) does the same on Android, talking to the same backend over the emulator bridge.

What I'd do next

The obvious extension is swapping the static guideline corpus for a versioned one, so updates to clinical recommendations propagate without a full re-ingest. I'd also want a proper held-out clinical validation set beyond UCI Cleveland — it's a well-studied benchmark, but it's not a substitute for prospective data.

02 / RESEARCH

PashtoSenti — Transformer-Based
Sentiment Analysis for Pashto

TIMELINE2026
TARGETACM TALLIP / IEEE Access
LICENSECC-BY 4.0 (planned)

A research proposal for the first benchmark dataset and NLP system built specifically for Pashto sentiment analysis — a language with 60M+ speakers and almost no NLP tooling.

The gap

Pashto sits in a strange spot: tens of millions of speakers, real social media volume, and essentially no sentiment analysis benchmark. Most multilingual NLP work skips it entirely because there's no labeled data to start from.

Proposed pipeline

Data collection across Facebook, Twitter/X, and YouTube, followed by Pashto-specific preprocessing — script normalization and custom tokenization, since standard multilingual tokenizers handle Pashto's orthography poorly. Three-class sentiment annotation with a Cohen's Kappa target of 0.70+ for inter-annotator agreement.

Benchmarking plan

Classical baselines — SVM, Naive Bayes, Logistic Regression — benchmarked against multilingual transformers (mBERT, XLM-RoBERTa) and a PashtoCorp-adapted XLM-R variant. Includes a cross-lingual transfer experiment from Urdu to Pashto, testing whether a related, better-resourced language can bootstrap performance.
